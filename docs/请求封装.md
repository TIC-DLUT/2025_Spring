## 为什么要封装 axios

假设你项目里有 20 个页面，每个页面都要请求后端接口。如果不封装，每个地方都得写：

```typescript
axios.post("http://127.0.0.1:8080/user/login", data, {
  headers: { "Content-Type": "application/x-www-form-urlencoded" },
});
```

问题来了：

1. 后端地址改了，你要改 20 个地方
2. 每个请求都要手动带上 token，忘带一个就 401
3. 每个请求都要手动处理错误，漏掉一个就没有提示
4. 代码重复，又臭又长

所以我们需要**封装**：把公共的逻辑提取出来，写一次，到处生效。

## axios 是什么

[axios](https://axios-http.com/) 是目前最流行的 HTTP 请求库。它可以运行在浏览器和 Node.js 中，用 Promise 封装了 XMLHttpRequest，用起来比原生 fetch 更方便。

它能做的事情：

- 发起 GET/POST 等 HTTP 请求
- 设置请求头、超时时间
- **添加请求拦截器和响应拦截器**
- 自动转换 JSON 数据

## 创建 axios 实例

```typescript
// src/utils/request.ts
import axios from "axios";
import { GET_TOKEN } from "@/utils/tokens.ts";

let request = axios.create({
  baseURL: import.meta.env.VITE_APP_BASE_API, // '/api'
  timeout: 5000,
  headers: {
    "Content-Type": "application/x-www-form-urlencoded",
  },
});
```

`axios.create()` 创建了一个独立的 axios 实例，可以给它设置默认配置：

- **baseURL**：所有请求的公共前缀。开发环境是 `/api`，会被 Vite 代理转发到后端 `http://127.0.0.1:8080`
- **timeout**：请求超时时间，5 秒内没响应就报错
- **headers**：默认请求头

## 请求拦截器

请求拦截器会在**每个请求发出之前**自动执行。我们用它来做两件事：

```typescript
request.interceptors.request.use((config) => {
  // 1. 注入 token
  const token = GET_TOKEN();
  if (token) {
    config.headers.token = token;
  }

  // 2. 将对象数据转换为 URLSearchParams
  if (
    config.data &&
    typeof config.data === "object" &&
    !(config.data instanceof FormData)
  ) {
    config.data = new URLSearchParams(config.data as Record<string, string>);
  }

  return config;
});
```

### 注入 token

用户登录成功后，后端会返回一个 token（JWT），我们把它存在 localStorage 里。之后每次请求，都自动从 localStorage 取出 token 放到请求头里。

后端的中间件会检查这个 Token 请求头，如果没有就返回 401。

### URLSearchParams 转换

这一步是本项目中**非常容易踩坑**的地方。

HTTP 请求发送数据有两种常见格式：

**JSON 格式**：

```
Content-Type: application/json

{"telephone": "13800138000", "password": "123456"}
```

**form-urlencoded 格式**：

```
Content-Type: application/x-www-form-urlencoded

telephone=13800138000&password=123456
```

后端 Go 的 Gin 框架中，`ctx.PostForm()` 方法**只能读取 form-urlencoded 格式**的数据，读不了 JSON。

但 axios 默认会把 JavaScript 对象序列化为 JSON。如果我们在 headers 中声明了 `Content-Type: application/x-www-form-urlencoded`，但实际发的还是 JSON，后端就**读不到数据**。

所以我们需要用 `new URLSearchParams(data)` 把对象转换成 form-urlencoded 格式：

```typescript
// 转换前
{ telephone: "13800138000", password: "123456" }

// 转换后（URLSearchParams 对象，发送时自动变成 form-urlencoded）
telephone=13800138000&password=123456
```

> 小提示：在实际开发中，如果前后端都是你写的，用 JSON 格式通信是最方便的。本项目使用 form-urlencoded 是因为后端用了 `ctx.PostForm()`。如果你以后自己写后端，建议统一使用 JSON 格式。

## 响应拦截器

响应拦截器会在**每个请求收到响应之后**自动执行：

```typescript
request.interceptors.response.use(
  (res) => {
    return res.data; // 成功：简化数据，直接返回 data 部分
  },
  (err) => {
    let message = "";
    const status = err.response.status;
    switch (status) {
      case 400:
        message = "参数出错";
        break;
      default:
        message = err.message;
        break;
    }
    return Promise.reject(new Error(message));
  },
);
```

成功时，`res.data` 就是后端返回的 JSON 解析后的对象（axios 自动解析了），我们直接返回它，这样调用方拿到的就是干净的数据，不需要再 `.data`。

失败时，根据 HTTP 状态码给出中文错误提示。

## 使用封装好的 request

封装完成后，在 api 层使用就非常简洁了：

```typescript
// src/api/user/index.ts
import request from "@/utils/request.ts";

enum API {
  LOGIN_URL = "/user/login",
  REGISTER_URL = "/user/register",
}

export const reqLogin = (data: LoginForm): Promise<LoginResponse> =>
  request.post(API.LOGIN_URL, data);

export const reqRegister = (data: LoginForm): Promise<RegisterResponse> =>
  request.post(API.REGISTER_URL, data);
```

注意，这里的 URL 是相对路径（`/user/login`），会自动拼上 baseURL（`/api`），变成 `/api/user/login`，然后被 Vite 代理转发到后端。

在组件中调用：

```typescript
const result = await reqLogin({ telephone: "13800138000", password: "123456" });
// result 就是 { code: 0, token: "xxx", message: "登录成功！" }
```

## 总结

封装 request 的好处：

| 不封装               | 封装后       |
| -------------------- | ------------ |
| 每次都要写完整 URL   | 只写相对路径 |
| 每次手动带 token     | 自动注入     |
| 数据格式需要每次处理 | 统一转换     |
| 错误处理到处写       | 统一处理     |
| 后端地址变了改 N 处  | 改一处       |

这就是工程化设计中最基础但也最重要的一步。
